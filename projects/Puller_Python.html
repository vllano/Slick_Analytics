Automated Log Pulling Project

Project Overview

This project details the successful planning, development, and deployment of a desktop application, "Puller". The goal was to significantly improve operational efficiency for the Field Operations team performing device prototype testing. This solution streamlines the post-collection process by automating the extraction and organization of logs from devices used in testing. By enabling the simultaneous transfer of multiple logs with user-defined naming conventions, "Puller" eliminates the time-consuming and error-prone manual methods of log retrieval and naming. This results in substantial time and effort savings, leading to improved operational efficiency and data accuracy.

Requirements Gathering

As a Business Analyst, I played a crucial role in analyzing the existing process and gathering user requirements to define a solution that streamlined the existing workflow. My approach included:

  • Workflow Analysis: Conducted a workflow analysis of the existing log collection process to identify bottlenecks and inefficiencies in the post-collection phase.
  • Feedback Gathering: Conducted interviews with Field Operators to understand their specific needs and pain points. This included understanding the variations in log retrieval methods across different device brands and models.
  • Documentation: Documented the requirements for the "Puller" tool, focusing on features to automate and streamline the log retrieval process.

Problem Statement

The core problems identified, which impacted the overall efficiency of the data collection process, were:

  • Time-Consuming Process: Manually pulling logs from multiple devices was a repetitive and time-intensive task.
  • Inconsistency: Variations in log retrieval methods across different device brands and models added complexity.
  • Error-Prone: Manual renaming and organization of logs increased the risk of errors and data mismanagement.
  • Lack of Standardization: The absence of a standardized process led to inconsistencies in log organization and naming conventions.

Design and Development

To address these challenges, I worked with stakeholders to define the following key requirements for the "Puller" application:

  • Automation: Automate the process of pulling logs from multiple testing devices simultaneously.
  • Standardization: Ensure a consistent process for log retrieval across different device models.
  • Efficiency: Reduce the time and effort required for log retrieval and organization.
  • Accuracy: Minimize manual steps to reduce the risk of errors in log naming and organization.
  • Flexibility: Allow users to define naming conventions for the retrieved logs.
  • Cleanup: Provide an option to delete files after the transfer process.

The "Puller" application was designed to satisfy the specified requirements and delivers the following core capabilities:

  • ADB Interaction:
    • Use the subprocess module to execute ADB commands like adb devices and adb logcat.
    • Implement functions to start and stop log capture.
    • Handle device connection/disconnection events.
  • Log Processing:
    • Create functions to parse raw logcat output.
    • Implement filtering logic based on priority, tag, and PID.
    • Implement search functionality using regular expressions.
  • Data Structures: Use appropriate data structures (e.g., lists, dictionaries) to store and manipulate log data efficiently.
  • Threading: Use the threading module to capture logs in a separate thread, preventing the GUI from freezing.
  • Customization Options: Enables users to automatically prefix log names with user-defined text.
  • Wiping Feature: Provide an option to delete files after transfer is complete.

Tech Stack

  • Programming Language: Python (primary)
  • GUI Framework: Tkinter
  • Libraries:
    • os (for interacting with the operating system)
    • shutil (for copying, moving, renaming, and deleting files and directories)
    • subprocess (for executing ADB commands)
    • threading (for managing log capture in the background)
    • Regular expression libraries
  • Packaging: PyInstaller

Data Flow and Implementation

The workflow can be broken down into the following key stages:

  • Initiation: The user launches the Puller application.
  • Device Connection: Puller establishes a connection with the testing device via ADB.
  • Device Verification: The application verifies that the connected device is authorized.
  • Log Selection: The user specifies the desired logs to be retrieved.
  • Log Transfer: Puller automates the transfer of selected logs to the user's local machine.
  • Log Organization: The application automatically organizes the retrieved logs into a structured folder system, renaming them for clarity.
  • Optional Cleanup: Puller can optionally deletes files after the transfer process.
  • Completion: The process concludes with a success or failure notification.

To ensure successful implementation, I:

  • Developed user documentation and training materials to facilitate user adoption by Field Operators.
  • Managed the deployment of the "Puller" tool to relevant users and provided ongoing support.

Results and Business Impact

The implementation of the "Puller" application resulted in:

  • Increased Efficiency: The application reduces log retrieval time by an average of 21 minutes per instance.
  • Improved Accuracy: Automation minimizes manual errors in log naming and organization.
  • Enhanced Productivity: Streamlining the log retrieval process allows Field Operators to focus on higher-priority tasks.
  • Standardized Process: The application ensures a consistent and standardized approach to log retrieval.
  • Reduced Operational Costs: Time savings translate to reduced operational costs.
